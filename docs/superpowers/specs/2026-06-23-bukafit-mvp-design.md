# BukaFit AI-тренер — дизайн MVP (v1)

> Дата: 2026-06-23. Язык продукта: русский. Этот документ описывает **КАК** строим
> (стек, архитектура, модули, данные). **ЧТО** и **ЗАЧЕМ** — в `BukaFit.md`.

---

## 1. Цель документа

Зафиксировать технический дизайн MVP перед написанием плана реализации. Масштаб
намеренно мал — 100–200 пользователей (пилот). Никакого оверинжиниринга.

---

## 2. Стек

| Слой | Выбор | Почему |
|------|-------|--------|
| Язык | Python 3.12 | Лучшая AI/LLM-экосистема, быстрый MVP |
| Telegram | aiogram 3 (async) | Современный async, FSM из коробки |
| ORM | SQLAlchemy 2 (async) | Зрелый async ORM |
| БД | PostgreSQL | Хватает с запасом на масштаб |
| Кэш/сессии | Redis | FSM-хранилище aiogram + горячий кэш (рычаг скорости) |
| Напоминания | APScheduler | Утренний план, вечерний прогресс |
| Миграции | Alembic | Версионирование схемы |
| Конфиг | pydantic-settings | Env-переменные, типобезопасно |
| Инфра | Docker Compose | app + postgres + redis одной командой |
| Тесты | pytest + pytest-asyncio | TDD |

**Telegram-режим:** long polling для локальной разработки (без публичного URL),
webhook для прода. Переключается флагом конфига. Старт — polling.

## 3. Архитектура: два слоя (BukaFit.md §7.1)

- **Скелет** — детерминированный код без ИИ. Команды, онбординг, логирование,
  напоминания, стрики, расписание, профиль, прогресс, прогрессия нагрузки.
  Это и есть продукт — работает сам.
- **ИИ-слой** — вызывается в 3 местах: генерация плана, ответы на вопросы,
  разбор еды (позже). Надстройка над скелетом, не замена.

**Подход: модульный монолит.** Один сервис, чёткие границы модулей. Не
микросервисы (док отвергает), не один файл (фичи спутаются).

## 5. Модель данных — компактная память (BukaFit.md §7.3)

Бот помнит пользователя, но **не хранит сырые диалоги для контекста**. Три части:

- **User** — `tg_id`, `created_at`, статус доступа.
- **Profile** (статичный) — цель (масса/сушка/здоровье), уровень, инвентарь
  (зал/дом), график (дни), травмы/ограничения, предпочтения. Маленький структ.
- **Program** — текущая недельная программа. JSON: дни → упражнения (название,
  целевые подходы/повторы/вес, замены).
- **WorkoutLog** — по упражнению: вес, повторы, RPE (ощущение нагрузки),
  сделал/пропустил, timestamp.
- **Summary** — rolling summary: текст ~500–1000 токенов, периодически уплотняется,
  `updated_at`.

Зачем: маленький контекст = быстрее, дешевле, без «забывания» важного.

## 6. Слой моделей (AI)

```python
class ModelProvider(Protocol):
    async def generate_plan(self, profile: Profile) -> Program: ...
    async def answer_question(self, question: str, memory: Memory) -> str: ...
    # async def analyze_food(self, image: bytes) -> FoodInfo: ...  # v2
```

- **MockProvider** — старт. Возвращает осмысленный, но захардкоженный план и
  ответы. Скелет полностью работает без API-ключей.
- **GeminiProvider / ClaudeProvider** — позже, тот же Protocol. Выбор через
  `config.AI_PROVIDER`. Логика бота не меняется.
- **«Живость» — это промпт, не модель** (BukaFit.md §8). Персона, список
  запрещённых клише, примеры тона, умеренная температура — в `prompts.py`.

---

## 7. Память для запроса (`ai/memory.py`)

Собирает компактный контекст для `answer_question()`:
1. Profile (статичный объект).
2. Состояние тренировок (текущая программа + последние веса/повторы).
3. Rolling summary (короткий текст).

Никаких сырых историй сообщений.

---

## 8. Ключевые потоки

1. **Онбординг.** FSM ведёт по шагам (цель → уровень → инвентарь → график →
   травмы) → сохраняет Profile → `generate_plan()` → сохраняет Program →
   показывает день 1.
2. **«Сегодня».** `schedule` выбирает день недели → показывает упражнения с
   предложенным весом/повторами из `progression` (по прошлому логу) →
   логирование inline-кнопками (сделал · ±вес · ±повторы · RPE) = 1-2 тапа.
3. **Чат.** Любой текст → `memory` собирает контекст → `answer_question()` →
   ответ + дисклеймер по здоровью. Индикатор «печатает…» + потоковая отдача.
4. **Напоминания.** APScheduler: утром — план на сегодня, вечером — фиксация
   прогресса и мотивация.

## 9. Прогрессия нагрузки (детерминированная, BukaFit.md §5.1 п.4)

Из последнего `WorkoutLog` по упражнению:
- Достиг целевых повторов **и** низкий RPE → предложить +вес или +повторы.
- Не дотянул / высокий RPE → держать вес или делоуд (снизить).
- Нет истории → стартовые значения из программы.

Простая таблица правил. Без ИИ.

---

## 10. Безопасность (BukaFit.md §5.3, §11)

- Узкий scope ИИ: техника, замена упражнения, базовое питание.
- Обязательный дисклеймер по здоровью в ответах.
- Никаких медицинских диагнозов/опасных советов.
- Белый список упражнений в программах.
- Эскалация к живому тренеру в спорных случаях.

---

## 11. Тестирование (TDD)

- Чистая доменная логика (`progression`, `schedule`, `streaks`) — юнит-тесты,
  без БД и ИИ.
- MockProvider делает ИИ-потоки тестируемыми офлайн.
- Хендлеры aiogram — с замоканным ботом.
- Репозитории — против тестовой БД (или транзакция с откатом).

---

## 12. Вне scope MVP (BukaFit.md §12)

Не делаем в v1: мобильное приложение, носимые устройства, видеоразбор,
авто-перестройка всей программы, полноценный модуль питания с калориями по фото,
биллинг-логику, реферальную механику. Реальные провайдеры Gemini/Claude
подключаются после готового скелета.

---

## 13. Метрика, на которую работаем (BukaFit.md §9)

Главная: доля пользователей, логирующих ≥2 тренировки в неделю на 1-й неделе.
Поэтому логирование — приоритет №1 по UX (1-2 тапа, дефолты, напоминания).
